# report-writing-guidelines.md

## 核心原则
最终交付物只有一个：`{arxiv_id}_阅读报告.md`

### 必须直接写入主报告的内容
- arXiv / hjfy / papers.cool 链接
- 主结果表
- 消融实验表
- 相关论文表
- 关键外部文献清单
- 图片与图片解释
- 关键公式理解卡

### 图片插入位置建议
- 任务图 / 数据流程图：`1.2 问题设定`
- 方法总览图 / 模块图：`2.3 核心推导与算法构造`
- 主结果可视化：`3.3 实验结果的解释力度`
- 失败案例：`3.4 潜在未讨论因素`
- 消融相关图：`3.3` 或 `3.4`

### 公式理解建议
- 优先解释 A级公式
- 每个关键公式至少给出：符号表 + 自然语言翻译 + 工程映射
- 最关键的 1–3 个公式尽量给 toy example

### 严禁
- 单独设置“图片区”把图集中堆放
- 把相关论文表只留在缓存 JSON
- 把主结果表、消融表只留在旁支文件
- 在最终报告中展示 hjfy / papers.cool 的状态说明行
- 只罗列公式而不解释


### 关键表格落地要求（v3 新增）
以下类型的表格，如果论文存在且被正文用于支撑结论，必须写进主报告：
- 主 benchmark 对比表
- 扩展指标表
- 数据规模 / scaling law 表
- 动作专家 / 解码器 / planner 对比表
- 关键消融表
- 泛化 / 额外实验表

若表格过宽：
- 可以拆成多个 Markdown 子表
- 也可以保留最关键列并注明删减
- 但绝不允许整张关键表缺失


### 公式解释就地插入要求（v4 新增）
- 不要单独设“公式理解卡专区”
- 公式解释要跟着上下文走
- 最佳模式是：方法段落里出现公式后，紧跟 1 段解释；如果还有方法图，再把图放在相邻位置
- 最理想的阅读体验是：读者在一个局部位置同时看到公式、图、表和解释
