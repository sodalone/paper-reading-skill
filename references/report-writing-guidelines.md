# report-writing-guidelines.md

## 核心原则
最终交付物只有一个：`{arxiv_id}_阅读报告.md`

### 必须直接写入主报告的内容
- arXiv / hjfy / papers.cool 链接
- 主结果表
- 消融实验表
- 相关论文表
- 关键外部文献清单
- 图片与图片解释
- 关键公式理解卡

### 图片插入位置建议
- 任务图 / 数据流程图：`1.2 问题设定`
- 方法总览图 / 模块图：`2.3 核心推导与算法构造`
- 主结果可视化：`3.3 实验结果的解释力度`
- 失败案例：`3.4 潜在未讨论因素`
- 消融相关图：`3.3` 或 `3.4`

### 公式理解建议
- 优先解释 A级公式
- 每个关键公式至少给出：符号表 + 自然语言翻译 + 工程映射
- 最关键的 1–3 个公式尽量给 toy example

### 严禁
- 单独设置“图片区”把图集中堆放
- 把相关论文表只留在缓存 JSON
- 把主结果表、消融表只留在旁支文件
- 在最终报告中展示 hjfy / papers.cool 的状态说明行
- 只罗列公式而不解释


### 关键表格落地要求
以下类型的表格，如果论文存在且被正文用于支撑结论，必须写进主报告：
- 主 benchmark 对比表
- 扩展指标表
- 数据规模 / scaling law 表
- 动作专家 / 解码器 / planner 对比表
- 关键消融表
- 泛化 / 额外实验表

若表格过宽：
- 可以拆成多个 Markdown 子表
- 也可以保留最关键列并注明删减
- 但绝不允许整张关键表缺失


### 公式解释就地插入要求
- 不要单独设“公式理解卡专区”
- 公式解释要跟着上下文走
- 最佳模式是：方法段落里出现公式后，紧跟 1 段解释；如果还有方法图，再把图放在相邻位置
- 最理想的阅读体验是：读者在一个局部位置同时看到公式、图、表和解释


### 表格就地插入要求
- 表格不能只出现在附录
- 只要正文某处分析依赖某张关键表，这张表就要在该分析附近出现
- 如果原表太宽，可以拆成精简子表放正文，完整表再补到附录 A
- 但不允许“正文讨论实验，读者却要翻到附录才能看到表格”

### 数学公式格式要求
- 行内公式统一写成 `$...$`
- 跨行公式统一写成：
  ```markdown
  $$
  ...
  $$
  ```
- 不要用反引号包公式
- 不要把公式写成普通文本“公式”


### 图片提取来源要求
- 优先从原始 PDF 提取图片
- 如果能提取嵌入图片，就不要用网页图片
- 如果需要裁图，应优先裁 PDF figure 区域，而不是网页区域
- 禁止把网页截图当成最终插图

### 公式编号与公式块分离
推荐写法：
- 正文：`式 (7) 定义了总奖励函数`
- 公式块：
  ```markdown
  $$
  ...
  $$
  ```

不推荐：
- 在公式块内部使用 `\tag{7}`


### 公式渲染兼容性要求
- 能单行写完的 display 公式，优先单行写
- 需要多行时，使用 `aligned`
- 不要在 display 公式中裸写行首 `+`
- 公式编号写在正文里，不写进公式块
- 如果已知目标环境不支持数学渲染，要补一行纯文本伪公式


### arXiv 源码图片优先策略
- 优先下载 `https://arxiv.org/src/{paper_id_with_version}` 源码包
- 优先使用源码中的原始 figure 文件，而不是 PDF 抠图
- 优先依据 `\includegraphics`、caption、label、section 来确定插图位置
- 若源码图片可用，就不要退回网页抓图或 PDF 抠图
